# Ralph Progress Log - Derby Digital Onboarding Portal

## Codebase Patterns
- Typography: font-heading (Anton, uppercase) for h1/h2 headings; font-sans (Inter) for body/labels; CSS vars --font-anton and --font-inter
- Input styling: h-12 height, rounded-lg, text-base, focus:shadow-sm — consistent across all form steps
- Label styling: text-sm font-semibold text-gray-700 mb-2 — consistent across all form steps
- Brand colors: derby-dark (#0A1628), derby-card (#1A2744), derby-blue (#2093FF), derby-blue-deep (#0026FF), derby-text (#1A1A2E)
- Light mode default: bg-gray-50 body, bg-white cards/inputs, gray-200/300 borders, gray-900 headings, gray-700 labels, gray-500 descriptions. No purple anywhere.
- New layout: vertical sidebar (320px) on left for desktop (lg:), horizontal mini-bar on mobile. Sidebar hidden on Welcome (step 1) and Confirmation (step 7)
- STEPS array now includes `icon` field (SVG path data) — used by VerticalStepTracker and potentially other components
- Root layout uses light mode (bg-gray-50, text-gray-900) — no more dark class on html element
- Step numbering in sidebar: displayed as "Step N-1" (Step 1 = Business Info) since Welcome is step 1 internally but not shown in tracker
- Use `@/components/wizard/WizardContext` for step state — useWizard() hook provides currentStep, formData, navigation
- API routes use `getServiceClient()` from `@/lib/supabase` for server-side Supabase access
- Session token stored in both WizardContext state and localStorage (`derby_session_token` key)
- Step components live in `src/components/wizard/steps/` — import into StepRenderer with conditional rendering by step number
- STEPS array and TOTAL_STEPS constant exported from WizardContext for step metadata
- Wizard animations use CSS keyframes in globals.css (animate-step-in, animate-step-in-reverse)
- Page is "use client" — WizardProvider wraps WizardLayout which wraps StepRenderer
- Mobile responsive: sm: breakpoint for desktop step indicators, mobile shows simplified "Step X of Y"
- PATCH /api/sessions/[token] merges form_data (server-side merge with existing data) — use for auto-save on blur
- Form validation pattern: validate on blur + validateAll on continue; errors stored in component state as FieldErrors record
- Phone formatting: strip non-digits, format as (XXX) XXX-XXXX using formatPhone() utility in each step component
- Auto-save uses debounced fetch (500ms) triggered on blur after field validation passes
- Service category cards use border-2 + bg-derby-blue/10 for selected state, with checkmark badge top-right
- "Other" service category conditionally shows a text input for specifying custom service type
- FormData stores other_service as separate field (not inside service_categories array)
- Use `&apos;` for apostrophes in JSX text — ESLint react/no-unescaped-entities rule enforced at build time
- Upload API (POST /api/upload) uses FormData with file, session_token, doc_type fields — returns doc metadata
- Document uploads go to Supabase Storage bucket 'documents' with path `{session_id}/{doc_type}.{ext}`
- Upload route uses upsert:true for re-uploads, deletes old documents row before inserting new one
- DocumentsStep tracks uploaded files in component state (not in formData) since docs are in separate table
- GET /api/documents?session_id=X returns doc_type + file_name for a session — use for review/detail displays
- POST /api/submit expects { session_token } — creates submissions row and marks session complete
- SUBMISSION_FIELD_MAP in types.ts maps formData field names to actual DB column names on submissions table
- Confetti animation: CSS class `animate-confetti-fall` with keyframe in globals.css — apply to absolutely positioned elements
- Admin API auth: Bearer token or `admin_token` cookie matching ADMIN_PASSWORD env var (default 'derby2026')
- GET /api/admin/submissions supports pagination (page, page_size), status filter, and search (ilike on business_name/contact_name)
- Admin pages live in `src/app/admin/` — layout.tsx has sidebar, login/page.tsx is the auth page
- Middleware at `src/middleware.ts` protects all `/admin/*` except `/admin/login` — checks `admin_token` cookie
- Admin login: POST /api/admin/login validates password, sets httpOnly `admin_token` cookie
- Admin layout skips sidebar for `/admin/login` pathname — check in layout component
- Logout: clear `admin_token` cookie via JS (`document.cookie = "admin_token=; path=/; max-age=0"`)
- GET /api/admin/submissions/[id] returns { submission, form_data } — form_data from sessions table has ad prefs not in submissions
- PATCH /api/admin/submissions/[id] accepts { pipeline_status, notes } — notes is jsonb column on submissions
- GET /api/admin/submissions/[id]/documents returns documents with signed_url from Supabase Storage (1hr expiry)
- Submission detail page at /admin/submissions/[id] — uses Section/Field/DocumentCard components
- Notes stored as `{ internal: "text" }` in submissions.notes jsonb column
- GET /api/admin/stats returns { total, new, in_progress, active, this_week, this_month, weekly_chart[] } — auth same as other admin endpoints
- WizardContext exposes `isRestoring` boolean — use it to show loading skeletons while session restore is in progress
- Admin pages pattern: always pair API fetches with both `loading` (for skeleton) and `error` (for friendly message) state variables
